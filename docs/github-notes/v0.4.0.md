# licit v0.4.0 — EU AI Act Framework

> licit now evaluates your project against the EU AI Act. 11 articles evaluated article-by-article with scoring, an interactive FRIA generator with auto-detection, and auto-populated Annex IV technical documentation — all from your project metadata.

## Highlights

- **EU AI Act evaluator**: 11 articles evaluated with dedicated scoring per article (Art. 9, 10, 12, 13, 14, 26, 27, Annex IV)
- **FRIA generator**: Interactive 5-step questionnaire with auto-detection of 8 fields from project context
- **Annex IV generator**: 6-section technical documentation auto-populated from project metadata
- **ComplianceFramework Protocol**: Extensible base for all compliance frameworks
- **497 tests** passing — mypy strict clean, ruff clean
- **8 functional CLI commands**: `init`, `status`, `connect`, `trace`, `changelog`, `fria`, `annex-iv`, `verify`

## What's new

### `licit verify --framework eu-ai-act` (fully functional)

```bash
licit verify --framework eu-ai-act    # Evaluate EU AI Act compliance
licit verify                          # Evaluate all enabled frameworks
```

**Example output:**
```
  Compliance Verification
  Compliant:     3
  Partial:       4
  Non-compliant: 4

  Non-compliant controls:
    [X] ART-9-1: Risk Management System
    [X] ART-14-1: Human Oversight
    [X] ART-14-4a: Human Oversight — Understand Capabilities
    [X] ART-27-1: Fundamental Rights Impact Assessment (FRIA)
```

Exit codes: `0` = all compliant, `1` = non-compliant, `2` = partially compliant.

### `licit fria` (fully functional)

Interactive 5-step questionnaire per EU AI Act Article 27:

```bash
licit fria            # Start new FRIA
licit fria --update   # Update existing FRIA
```

**5 steps, 16 questions:**
1. System Description (purpose, technology, models, scope, human review)
2. Fundamental Rights Identification (data, employment, security, discrimination)
3. Impact Assessment (risk level, max impact, detection speed)
4. Mitigation Measures (guardrails, scanning, testing, audit trail)
5. Monitoring & Review (frequency, responsible person, incident process)

**Auto-detection** for 8 fields: system_purpose, ai_technology, models_used, human_review, guardrails, security_scanning, testing, audit_trail. Auto-detected values are shown with a confirmation prompt.

**Output:**
- `.licit/fria-data.json` — Raw responses (reusable with `--update`)
- `.licit/fria-report.md` — Rendered Markdown report via Jinja2 template

### `licit annex-iv` (fully functional)

```bash
licit annex-iv --organization "ACME Corp" --product "WebApp"
```

Auto-populates 6 sections from project metadata (27 template variables):

| Section | Content |
|---|---|
| 1. General Description | Purpose, AI components, languages, frameworks |
| 2. Development Process | Git stats, provenance %, agent configs |
| 3. Monitoring & Control | CI/CD, audit trail, changelog |
| 4. Risk Management | Guardrails, quality gates, budget, oversight, FRIA |
| 5. Testing & Validation | Test framework, security tools |
| 6. Changes & Lifecycle | Tracking mechanisms summary |

Sections missing evidence generate actionable **recommendations** (e.g., "Run `licit trace` to begin tracking code provenance").

### ComplianceFramework Protocol

New extensible base for all compliance frameworks:

```python
@runtime_checkable
class ComplianceFramework(Protocol):
    @property
    def name(self) -> str: ...
    @property
    def version(self) -> str: ...
    @property
    def description(self) -> str: ...
    def get_requirements(self) -> list[ControlRequirement]: ...
    def evaluate(self, context: ProjectContext, evidence: EvidenceBundle) -> list[ControlResult]: ...
```

`EUAIActEvaluator` implements this protocol. Phase 5 (OWASP) will add a second implementation.

## QA hardening

2 issues found and fixed during QA:

- **(Medium)** FRIA template used direct dict access `{{ responses.responsible_person }}` instead of safe `.get()` — could raise UndefinedError in strict Jinja2 mode
- **(Medium)** Annex IV template had excessive blank lines between sections due to Jinja2 whitespace handling — fixed with `{%-`/`-%}` whitespace control
